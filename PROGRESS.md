# Progress

Use this file to capture concise, durable insights when a task produces knowledge worth preserving. Organize entries by topic or subsystem rather than timeline, and preserve reusable repo knowledge, implementation decisions, caveats, and verification notes; skip routine task logs.

## Repository Practices

- `PROGRESS.md` is for judgment-based durable insights worth preserving, not every task.
- Organize `PROGRESS.md` by topic or subsystem instead of chronological task history.
- Root threads use the built-in root prompt from `docs/system_prompt.md`, selected by `SystemPromptKind::Root`; `SMOOTH_CODE_LLM_PREAMBLE` replaces that root prompt only for root threads. Subagents ignore that env override and use their tool-owned built-in prompt.
- `review-codes` is a project-local skill at `.smooth-code/skills/review-codes/SKILL.md`; it is review-only by default, scopes from explicit user targets or working-tree diffs, and reports findings by severity with file references plus verification status.
- `commit-changes` is a project-local skill at `.smooth-code/skills/commit-changes/SKILL.md`; it is for user-requested commits, inspects/stages the intended scope explicitly, and requires a multi-line commit message with an imperative summary plus concrete body details and validation/caveats.
- Environment context placeholders in the selected base preamble are filled by `crates/core/src/environment.rs` before plan-mode text is appended. Keep this context cache-stable: working directory, git-repo yes/no, platform, OS version, shell, and preferred CLI availability (`rg`, `fd`, `eza`).
- `docs/system_prompt.md` intentionally emphasizes honest uncertainty: state limits or mistakes plainly, avoid guessing, and ask the user before proceeding when intent or a required decision cannot be safely inferred.
- The model-facing `list_dir` and `dynamic_echo` tools were removed; directory listings should go through shell commands such as `eza`, and client-mediated interaction should use the typed `ask_user_question` tool/server request. Plan mode includes `run_command` for read-only exploration/validation, but still withholds file-mutating `delete`, `edit`, and `write`.
- Source changes should go through structured file tools (`edit`, `write`, `delete`) so the runtime can emit structured `FileChangeOutput`; keep `run_command` for inspection, validation, formatters, and project commands rather than Python/`sed -i`/`awk`/redirection rewrite scripts.
- `run_command` output truncation is byte-capped but must truncate only at valid UTF-8 character boundaries before appending the truncation marker; command output can include CJK/emoji/non-ASCII paths and must not panic inside `String::truncate`.
- The `edit` tool keeps the public tool name but now uses a typed sed-style literal replacement API for one existing UTF-8 file per call: `file_path`, ordered `replacements[]` entries with `old_string`, `new_string`, optional `replace_all`, and optional `move_to`. The old hunk-based `updates[].hunks[].lines[]` schema is intentionally rejected. `old_string` is literal text (not regex), may span lines, must be non-empty, and must match exactly once unless `replace_all: true`; empty `new_string` deletes text. Empty `replacements` are allowed only for a real move, self-move targets normalize to no move, no-op edits fail, and edits prepare all replacements before mutating files; apply-phase filesystem failures are not transactionally rolled back.
- Plan-mode tool gating must happen at agent build time: rig-core 0.37 bakes tool definitions and the preamble into the `Agent` when it is constructed, with no per-request filtering seam, so hiding `edit`/`write`/`delete` from the model in plan mode requires a separate agent. Both variants are therefore prebuilt at thread creation (`SessionModels` on `Session`) and plan-mode toggles are a flag flip plus `PlanModeChanged` — never a rebuild. This also keeps provider state (e.g. the parked OpenAI WebSocket) alive across toggles and removes the old rebuild-time env re-read/cwd bugs.
- `exit_plan_mode` is intercepted in the turn loop and gated on user approval via the `RequestPlanApproval` server request; the plan file written by `plan_write` (path from `tools::plan_file_path`) is the approval source of truth, not tool-call args. Rejection is a *successful* tool result carrying the user's feedback so the model revises instead of treating the call as errored. The approval wait races the turn cancellation token, and turn interrupt also fails the pending request through `abort_pending_server_requests`.
- When approval flips plan mode off mid-turn, the rest of the turn streams from the *normal* session model. Stub-based tests must model this: a turn-sequencing driver keyed by call count will silently restart from call 0 after the flip because the plan and normal models are distinct driver instances (see `crates/core/tests/plan_mode_e2e.rs`).
- Plan-mode spawns are coerced to the structurally read-only `Explore` prompt kind at `start_spawn_tool_call`; `PLAN_MODE_TOOLS` in `agent/plan_mode.rs` is the single source of the allowed-tool list (instructions are generated from it, a provider test pins the registry to it).
- `AskUserClient` is a concrete cloneable callback wrapper shared by root, resumed, and spawned child threads. `ask` routes from `AskUserQuestionParams.thread_id` to avoid duplicate thread identity at the call boundary; `abort_pending_server_requests` still takes `ThreadId` for cancellation. No separate ask-user client factory or thread-scoped outgoing sender wrapper is needed. Keep the stored client on `Core::Session`; `CoreThread` should not duplicate it.
- `AskUserQuestionParams` carries only `threadId`/`turnId`/`questions` (wire shape pinned by a protocol round-trip test). The old `callId` was removed: it was a tool-minted UUID that could never correlate with transcript tool rows, which are keyed by the agent-loop call id from `ToolCallStarted/Completed`. `ask_user_question` formats its tool result as one `"<question>": <answers>` line per question (plus optional preview block); the TUI never renders `output_preview`, so the user-visible Q→A record is the client-side transcript row, not the tool result.
- `app-server-protocol` imports `tokio::sync::oneshot`, so it must enable Tokio `sync`; with this repo's `tokio_unstable`/Tokio tracing setup it also needs `rt` to compile as a standalone package.

## Project Instructions

- Root thread start loads project-local instructions in the TUI, scoped by the nearest upward `.git`; with no git root only the current directory is scanned. Directories are scanned root-to-cwd, `AGENTS.override.md` wins over `AGENTS.md` per directory, empty files are skipped, invalid UTF-8 is decoded lossily with a warning, and loaded file text is capped at 32 KiB total on char boundaries before core adds model-facing wrapper text. Instruction file reads are bounded to the remaining budget plus a small UTF-8/truncation probe, so oversized files are not fully read into memory.
- Project instructions are injected Codex-style as a synthetic user-role context message on each provider request, before persisted history and before the live turn messages. This keeps AGENTS at a stable index-0 request prefix for provider prefix caching, avoids the OpenAI WebSocket system-role input problem, and keeps the context out of transcript events, `initial_messages`, persisted conversation history, and thread-list `last_user_message`. `source_path` is persisted metadata for resume/debug; the model-facing block header names the scoped directory.
- The loaded instruction snapshot is stored in rollout `SessionMeta` as optional `projectInstructions` and restored into `ResumeState`, so resumed root threads keep the original AGENTS context without re-reading files. Spawned subagents start with no inherited project instructions unless a future explicit start path supplies them.

## Error Handling

- Workspace crates deny `clippy::unwrap_used` and `clippy::expect_used` across all targets, including tests. Use `Result`-returning tests, `?`, explicit `let Some/Ok ... else { panic!(...) }` assertions, or typed error conversion instead of `unwrap`/`expect`.
- Wire-level errors are structured: `app-server-protocol::JsonRpcError` keeps JSON-RPC `code`/`message` and carries `smooth_protocol::ErrorInfo { kind, message, source, details }` in `data`. Protocol error events and errored agent statuses also carry `ErrorInfo` directly.
- Crate boundaries use typed errors: `smooth_protocol::AgentPathError`, `smooth_core::CoreError`, `app_server::AppServerError`, `tools::ToolError`, and `smooth_tui::TuiError`. Keep `anyhow` at entrypoints/tests or provider-facing glue, but prefer crate result aliases for internal public APIs.
- `AgentPath` parsing/join/resolve errors are typed and serde validation still uses the same string representation; path-bearing protocol structs should rely on `AgentPath` serde rather than accepting unchecked strings.
- App-server JSON-RPC conversion is centralized through `AppServerError::to_json_rpc_error`; core errors preserve their `smooth-core` source and typed kind when surfaced to clients. Tool errors expose stable `kind()` values while preserving readable `Display` text for model/UI output.

## OpenAI Provider

- The OpenAI provider uses Rig's Responses WebSocket session for the manual tool loop; keep OpenAI turn streaming on `stream_completion_turn` and do not route OpenAI through the generic `CompletionModel::stream()` SSE path.
- OpenAI WebSocket mode reuses one per-thread socket across sequential stateless `response.create` requests and reconnects after poisoned sockets or retryable early failures. This is safe only while smooth-code sends full-history stateless requests with no `previous_response_id`, `conversation_id`, `store`, or connection-scoped routing state; revisit reuse if that changes.
- Defensively handle a reused OpenAI WebSocket receiving a trailing `response.done` after an earlier primary terminal frame (`response.completed`/`failed`/`incomplete`). Park the prior terminal response ID with the socket and skip only a matching late `response.done` on the next request, or it can terminate the next turn with stale metadata.
- Connection-limit and early-close OpenAI WebSocket failures are retryable only before any visible assistant output has been yielded, so retries cannot duplicate streamed text or tool calls.
- Retryable OpenAI WebSocket resets after visible assistant output are handled in the regular task consumer, not by provider request replay: the partial assistant message is folded into `new_messages`, a transient `StreamError` notice is emitted, and the next manual-loop iteration rebuilds the stateless full-history request so the model continues from the partial turn. Per-attempt assistant item IDs (`base`, then `base#N`) keep the TUI from appending continuation text onto the finalized partial. Stub tests validate plumbing/history/tool execution; seamless Responses continuation still needs a manual proxy smoke test because the real API may restart from the partial assistant message.
- WebSocket item decoding in `crates/core/src/provider.rs` preserves provider item IDs for reasoning deltas and reuses one internal call ID across tool-call name, args, and final tool-call events so the TUI/core can correlate streamed tool state.
- The local OpenAI-compatible proxy can close/reset WebSocket sessions without a closing handshake; retry only before any assistant item is yielded, and only treat a reset as graceful after a complete output item or terminal response has been observed.
- A parked/reused WebSocket may also fail before output with `IO error: Connection reset by peer (os error 54)` after a long tool/subagent wait; classify that as the same retryable OpenAI WebSocket reset so the provider reconnects instead of failing the parent turn.
- The local OpenAI-compatible proxy can also fail during WebSocket startup with early close/no-close-reason messages, the generic `An error occurred while processing the request.` provider error, or explicit `stream closed before response.completed` / `disconnected before completion` error events emitted when the upstream Responses stream drops. These are retryable only before any assistant item is yielded, to avoid duplicating visible output or tool calls.
- The local OpenAI-compatible WebSocket proxy closes immediately after upgrade when `input` contains `role: system`; after Rig builds the Responses request, move system-role input items into the provider-level `instructions` field instead of downgrading them to user-visible text.
- Keep OpenAI WebSocket inbound parsing local and tolerant rather than relying on Rig's private `StreamingCompletionChunk` parser; the local proxy emits provider telemetry and some known event types with payload shapes that may not match Rig's strict structs.
- OpenAI WebSocket lifecycle responses from the local proxy may omit `output`; parse lifecycle fields (`status`, `usage`, errors, optional output IDs) from raw JSON instead of deserializing the full Responses completion object.
- OpenAI Responses requires assistant `message` input items that came from a reasoning response to be paired with the corresponding `reasoning` input item (`msg_*` depends on `rs_*`). If a fallback parser extracts only the assistant message ID/text from a terminal `response.completed`/`response.done` `output`, it must also extract the paired reasoning output or drop the provider message ID before replaying history.
- The OpenAI WebSocket accumulator parses terminal `response.output` items and defers fallback reasoning until `finish()`, emitting reasoning before the assistant `MessageId`; this preserves `rs_*`/`msg_*` pairing when the proxy only supplies reasoning in terminal lifecycle output.

## Configuration (smooth-config)

- Layered TOML config lives in the leaf crate `smooth-config` (package `smooth-config`, lib `smooth_config`); it depends only on serde/toml/thiserror/etcetera so `tools`, `core`, `app-server`, and `tui` can all use it without breaking layering. Precedence low→high: built-in defaults → `~/.config/smooth-code/config.toml` (etcetera `Xdg::new()` explicitly, so `~/.config` on macOS too) → `<cwd>/.smooth-code/config.toml` → legacy `SMOOTH_CODE_*` env vars. See `docs/config.md` and `config.example.toml`.
- Merge model is the two-type pattern: each layer parses into `PartialConfig` (all-`Option`, `#[serde(default, deny_unknown_fields)]`), layers merge highest-wins (nested sections recurse via `Merge`, never wholesale-replace), then `PartialConfig::resolve()` folds onto `Config::default()` and validates. `Config::default()` is the single source of truth for the built-in layer, and every default reproduces the prior hardcoded value exactly (e.g. openai `cazean`/`localhost:8317`, reasoning `high`, 300s command timeout, `CatppuccinMocha`).
- Load happens **once** in `crates/tui/src/main.rs` before telemetry and is threaded as `Arc<Config>` through `run` → `AppServerClient::start` → `InProcessStartArgs::new` → `MessageProcessor`/`CoreMessageProcessor` → `ThreadManagerState::new_with_config`. App-server/core never load config themselves (preserving the single-load guarantee); `ThreadManagerState::new` is a 2-arg test convenience that delegates with `Config::default()`.
- Provider wiring replaced `SessionModel::from_env` with `SessionModel::from_config` and `EnvSessionModelFactory` with `ConfigSessionModelFactory` (the exported default; swapped in `core/src/lib.rs`). Critically, `ThreadManagerState::new_with_config` resolves the factory **once** into a concrete `Arc<dyn SessionModelFactory>` and stores it, so spawned subagents (via `AgentControl`) inherit the same configured factory instead of falling back to a default. The root-only preamble override in `compose_session_preamble` is preserved.
- Env overlay is a pure function over an injected iterator (`env_overlay`), so tests pass a local map rather than mutating process-global env. Env vars stay permissive (invalid/zero → fall through); file values are strict. Only the five legacy vars are mapped — no generic per-key env layer in v1. Provider HTTP API keys stay env-only (not in the file schema).
- Tool limits are injected via builder methods (`ReadTool::with_default_limit`, `RunCommandTool::with_limits`, `TodoWriteTool::with_max_todos`, `{Delete,Edit,Write}Tool::with_max_file_change_bytes`) chained in `provider::build_agent(&Config)`; `::new(cwd)` keeps the default so the ~30 test construction sites are untouched. Model-facing `definition()` text for `run_command`/`read` is built from the configured values so the model isn't told stale defaults. Agent depth/threads moved onto `AgentControl::with_limits`.
- TUI appearance reads a process-global `Arc<Config>` installed once in `run()` (`crates/tui/src/config_state.rs`), since the render path has no injection seam. The global is replaceable and poison-safe (`current()` recovers via `into_inner`, never `unwrap`) so appearance tests aren't order-dependent. `highlight.rs` caches themes by name (`OnceLock<RwLock<HashMap<String, Arc<Theme>>>>`) instead of a single `OnceLock<Theme>`, so reinstalling config takes effect. Theme name → two-face `EmbeddedThemeName` mapping + startup validation (`TuiError::Config`) live in the TUI, keeping `smooth-config` two-face-free; `CatppuccinMocha` is the defensive fallback. `ColorSpec` (config) converts to `ratatui::Color` via `config_state::to_color` (free fn; orphan rule precludes a `From` impl).
- **Full surface, incl. the two once-deferred knobs:** `[provider.websocket]` (retry budget/backoff/cancel-drain) is carried on `OpenAiSessionModel.websocket`; the provider stream reads it directly, and the manual turn-retry loop (`tasks/regular.rs`) reads it via `SessionModel::websocket_retry_budget()` / `websocket_retry_delay()` (which return defaults for non-OpenAI/stub models so the marker-gated retry path keeps its historical behavior). `tools.max_skill_bytes` is carried on `AgentControl` (the config-threaded handle reaching root + subagent creation, already holding the agent limits), so `regular.rs` reads `session.agent_control.max_skill_bytes()`, `SkillTool` gets it injected in `build_agent`, and the TUI popup reads `config_state`. `provider.provider` is normalized through a single shared `smooth_config::normalize_provider` (trim + lowercase) used by both validation and `from_config`, so a value that passes validation can't later fail provider matching.

## Rollout and Resume Persistence

- Rollout `HistoryMessage` supports a full Rig `Message` variant; new user/assistant history records should use it so assistant IDs and structured reasoning, including encrypted/redacted provider payloads, survive resume. The legacy `UserText` and `AssistantText` variants remain readable for older sessions.
- Completed tool-loop turns must persist the full model-facing message tail after the already-recorded initial user prompt, including assistant tool calls, user `ToolResult`s, and subagent completion text messages. Persisting assistant messages alone leaves resumed provider history with unmatched tool calls.
- `Session::persist_history_messages` is fallible: it returns `false` if any rollout append fails (logging a warning) so callers can skip irreversible follow-up. The manual turn loop uses this to close consumed-subagent parent→child edges only after the turn's model-facing tail — which includes every consumed child's completion — is durably persisted, and never on the cancel/early-return/persist-failure paths. This guarantees an edge is never closed before the result that supersedes it is durable; an interrupted, crashed, or unpersisted turn leaves the edge open and resume reaps the finished child. Note that `Session::emit_event` persistence stays best-effort and still ignores rollout append errors, so do not gate irreversible state on an event having been persisted.
- Thread-list `last_user_message` previews should come from rollout user-message event records, not model-facing full user messages; full user messages may be internal tool/subagent context such as `agent_completed` JSON.
- `EventMsg` is internally tagged (`type`), so variants that cross app-server/TUI JSON boundaries must serialize as object payloads. Keep text-only `UserMessage` and fallback `AgentMessage` as struct variants with `text`; primitive newtype variants fail `ThreadResumeResponse.initial_messages` serialization.
- `AgentReasoningCompleted` is persisted and replayed on resume for visible reasoning transcript rows. Raw reasoning deltas remain transient stream state.
- Regular turns always use the manual `stream_completion_turn` path. The old opaque `stream_turn`/`SessionStream` provider surface was removed; test stubs should implement `SessionModelDriver::stream_completion_turn` and include a `SessionTurnSummary.response` for final assistant text.

## TUI Subagent Display

- Subagents are model-facing only through `spawn_agent`, with required `description` and `prompt` args plus optional `subagent_type`; omitted, `default`, and `general-purpose` select the implementation-capable default child, while `Explore`/`explore` select the built-in read-only explorer. Explore children start with empty history, receive `prompt` as their input, use `SystemPromptKind::Explore`, and expose only `read` plus `run_command`.
- Spawned subagents always start with empty history. Do not reintroduce `fork_context` or parent rollout-history seeding unless the spawn lifecycle and replay contract are deliberately redesigned.
- The parent TUI does not subscribe to child thread streams or render a child transcript. Child-visible output reaches the parent as collaboration lifecycle events emitted on the parent thread (`CollabAgentSpawnBegin`, `CollabAgentSpawnEnd`, `CollabAgentCompleted`).
- Subagent preview is available from transcript-select mode with `gd` on a `spawn_agent` tool row. The row becomes enterable as soon as core emits `CollabAgentSpawnEnd.new_thread_id`, before any `ToolCallCompleted` status/final result; `ToolCallCompleted.related_thread_id` remains the persisted/replay path and fallback. The TUI stores the child thread on `ToolCallGroupCell` and displays a dim `↳ subagent` suffix.
- `spawn_agent` lifecycle events must use the same internal tool call id as `ToolCallStarted`/`ToolCallCompleted`, not a separate UUID; otherwise the TUI cannot correlate `CollabAgentSpawnEnd` with the live tool row and `gd` reports "Subagent not started yet" while the child is already running.
- Subagent previews are read-only full-screen stacked views (`crates/tui/src/subagent_preview.rs`) opened through app-server `threadPreview`/`threadUnwatch`. Live previews subscribe before snapshotting the rollout, so duplicate persisted events are expected and must be deduped by call/item IDs; active streamed tails are temporary and completed full-text events win so mid-stream joins do not truncate messages.
- `ThreadManagerState::preview_thread` is intentionally read-only and must not reuse `resume_thread`: it snapshots rollout state without registering a root or resuming children. Live previews use rollout `RecoveryMode::PreviewLive` to avoid synthesizing `resume_recovery` interruption for an open turn; root preview status is folded chronologically from replayed events so a later open turn reports `Running` and a later error reports `Errored`.
- App-server preview subscriptions are ownership-tracked with a root flag plus preview watcher refcount. Each successful preview push sends exactly one `threadUnwatch` when popped/cleared; abort the forwarding task only when no root owns it and the preview count reaches zero. Server request overlays from previewed children are accepted after closing the preview stack so the overlay can receive keys.
- A live subagent tool result is emitted as `ToolCallCompleted` with `result_kind: StatusUpdate` and `related_thread_id`; the TUI records that child-thread-to-call mapping and keeps the tool row running until the matching `CollabAgentCompleted` arrives.
- `CollabAgentSpawnBegin` and `CollabAgentSpawnEnd` are transcript-silent in the TUI because the `spawn_agent` tool row already displays the prompt/arguments and running state; keep prompt/status text out of extra info rows unless adding a distinct subagent transcript surface.
- `CollabAgentCompleted` is transcript-silent in the TUI; it only finalizes the correlated `spawn_agent` tool row. The detailed child result for the model is separate structured JSON returned by the manual tool loop.
- `CollabResumeBegin` and `CollabResumeEnd` are also transcript-silent in the TUI. Core can emit them during resume hydration to re-open child subtrees, but the parent transcript should not show "Resuming agent" or child final-message status rows.
- Parent completion notifications should include every final child status, including `Shutdown` and `NotFound`, so retained inline waiters and TUI `spawn_agent` rows cannot remain running after a child reaches a terminal state.

## Turn Cancellation

- User-facing cancellation flows through `ClientRequest::TurnCancel`/`TurnCancelResponse`; `ThreadManagerState::cancel_turn_subtree` now delegates to `Core::interrupt_turn_cascade` — the same path `Op::Interrupt` takes — which aborts the root's turn and every live descendant's and returns only thread IDs whose active turn was actually interrupted. Idle/completed threads keep their existing `AgentStatus`.
- In the TUI, Ctrl-C cancels only while `is_turn_running`; idle Ctrl-C still exits. Esc in workspace Normal mode is a no-op for running turns; user-facing interrupt is reserved for Ctrl-C and `:cancel`. Overlay dismissal keeps priority structurally because pickers/plan-approval dispatch before mode handling, and Insert-mode Esc still just returns to Normal. `:cancel` uses the same effect path, and `TurnInterrupted` clears any ask-user overlay because core aborts pending server requests for the interrupted turn.
- Cancellation is two-phase with **graceful-with-deadline** semantics. Phase 1 (`Session::abort_all_tasks(CancelReason)`) is the immediate feedback path: cancel tokens, run abort hooks, emit `TurnInterrupted`, record the reason in `Session::last_cancel_reason`, and return the drained `RunningTask`s instead of dropping them (an immediate drop would hard-abort via `AbortOnDropHandle` and cut cooperative cleanup short). Phase 2 (`drain_aborted_tasks`) drains all tasks **concurrently against one shared deadline** — `CancelReason::drain_grace()` (8s for user-interrupt/replaced, 3s for shutdown/closed) is a per-batch window, not per-task, so N stubborn tasks cost one grace; each drain registers the task's `done` notify before the `is_finished` check to close the race, then hard-aborts at the deadline. Each cancellation path collects its own tasks plus the descendant cascade's into a single batch before draining: `Op::Interrupt` spawns that one drain so the cancel response returns immediately; turn replacement and shutdown await it inline — the old task may still be writing history/rollout and the new turn must not interleave.
- The task runner suppresses a second `TurnInterrupted` if it later observes a cancellation token after the active turn was already drained. When a runner finishes normally, remove its `RunningTask` from `active_turn` before terminal status publication so late cancellation cannot drain it, but leave an empty active-turn barrier until after the terminal event is emitted so the next turn cannot overtake it.
- The per-turn token reaches tools via a `tokio::task_local!` in the `tools` crate (`with_tool_cancel_scope` / `tool_cancel_token`): rig 0.37 awaits `Tool::call` inline in the caller's future, so `execute_normal_tool_call` scopes each call with a `child_token()` of the turn token (a per-call child can never leak across a turn switch). Outside a scope the ambient token degrades to never-cancelled. `run_command` is fully async (`tokio::process`, `process_group(0)`, `kill_on_drop`) and races the ambient token plus a per-call timeout (`timeout_secs` arg, env default `SMOOTH_CODE_RUN_COMMAND_TIMEOUT_SECS`, 300s). On cancel/timeout it returns immediately after SIGTERMing the group and scheduling an **unconditional** SIGKILL sweep (2s later) through the kill-sweep registry (`tools::kill_sweep`). Unconditional because the shell exiting does not empty the group (a SIGTERM-ignoring grandchild survives it; a dead group is a harmless ESRCH). The registry exists because each owner lifetime proved insufficient in turn: a detached task survives the tool future being dropped and the turn task being hard-aborted, but not the *process* exiting — and shutdown deliberately finishes faster than the sweep grace. Exit paths (`shutdown_all`, the TUI epilogue after its shutdown timeout) call `tools::sweep_pending_process_kills()` (re-exported from `smooth-core`) to fire still-pending SIGKILLs immediately; registry removal decides which side kills, so a pgid is never SIGKILLed twice across the grace window.
- A cancelled tool batch is **drained, not dropped**: in-flight futures get `tool_cancel_grace(reason)` to resolve — `CancelReason::tool_drain_grace()` (5s for user-interrupt/replaced, 1.5s for shutdown/closed), kept strictly under the same reason's `drain_grace()` so placeholders and persistence always land before the hard abort; the `SMOOTH_CODE_TOOL_CANCEL_GRACE_MS` override is clamped to `drain_grace() − 1s` so no configuration can push the batch past the deadline (run_command's detached kill means the tool resolves instantly on cancel, so even the short shutdown budget is comfortable). Anything still outstanding is completed with a placeholder `ToolCallResultKind::Interrupted` result so every call in the batch ends with a `ToolCallCompleted` event and a matched tool result (providers require a result per call). The cancelled turn is then finalized via `finalize_interrupted_turn`: `persist_turn_tail` + `replace_history`, deliberately **without** `close_consumed_child_edges`, so the model sees its partial output and the interrupted tool results on the next turn while resume reaping stays intact. `finalize_interrupted_attempt_and_turn` additionally folds uncommitted attempt output (partial text, never-executed buffered tool calls) — only valid on paths that have not run `commit_attempt_messages` for the attempt, or text would commit twice.
- Internal cancellation paths cascade to live descendants via the registry walk (`AgentControl::abort_descendants`/`abort_threads`, phase 1 only — the caller merges the returned tasks into its own drain batch), **not** hierarchical tokens — children are session-scoped and retained subagents legitimately outlive the spawning turn. `Op::Interrupt`/`Op::Shutdown` cascade unconditionally; turn replacement cascades only when a running task was actually replaced, so an idle session starting a fresh turn leaves retained subagents alone. Spawn starts remain uncancellable mid-start; `dispatch_tool_calls` re-checks the token right after the starts and phase-1 aborts just-started children the earlier cascade walk could not have seen, carrying `Session::last_cancel_reason()` for the right label/status — but their drain is **always spawned, never awaited**: that code runs inside the turn task being drained under the same deadline, and blocking on a stubborn child there would eat the parent's grace and cost it the persistence the grace protects.
- Provider-side cancel: the OpenAI WS stream takes the turn token (`SessionModel::stream_completion_turn(_, _, cancel)`); on cancel it sends a `{"type":"response.cancel"}` frame and keeps consuming (without yielding) under a 1.5s drain deadline — a clean terminal parks the socket for the next turn, a deadline expiry falls back to today's drop-the-socket. The turn loop keeps the cancelled stream polled (`drain_cancelled_stream`, gated on `SessionModel::requires_stream_cancel_drain()` so stub/HTTP providers skip the wait) because the WS cancel runs inside the stream. HTTP providers cancel by drop.
- Process exit is a single epilogue in `tui::run()`: every break path (plus SIGINT/SIGTERM via `TerminateSignals` — raw mode delivers interactive Ctrl+C as a key event, so signals only matter for external kills) runs `ClientRequest::Shutdown` → `ThreadManagerState::shutdown_all` (Op::Shutdown per root thread, cascading and killing tool process groups) bounded by a 5s timeout, then restores the terminal.
- Cancellation tests that need a child to finish should wait on the child's status watcher until a final `AgentStatus` instead of sleeping for a fixed duration; otherwise slow CI can turn an idle-cancel assertion into a legitimate active interrupt.

## Turn Loop

- The manual turn loop (`crates/core/src/tasks/regular.rs::run_manual_turn`) is split into an `AttemptState` (per provider-stream-attempt accumulation: `pending_tool_calls`, `accumulated_text`, `accumulated_reasoning`, `pending_reasoning_deltas`, the two `saw_*_this_attempt` flags, terminal `turn_summary`) and turn-scoped driver state that stays in `run_manual_turn` (`new_messages`, `completions_by_index`, `saw_tool_loop`, `retained_subagents`, `consumed_children`, `stream_retries`, `stream_phase_completions`). A fresh `AttemptState` is created per `'turn_loop` iteration — the initial attempt plus each mid-stream-retry continuation. Do not move the turn-scoped fields into `AttemptState`; `stream_phase_completions` in particular is turn-scoped retained-completion delivery state, not per-attempt.
- `AttemptState::ingest_event` folds one streamed `SessionCompletionEvent` into the attempt and emits the live delta / reasoning / tool-started events. `saw_tool_loop` is turn-wide (it drives the empty-`String` return when a turn ran tools but ended with no assistant text), so the driver ORs it from `attempt.saw_tool_call_this_attempt` after each event rather than letting the per-attempt method own it. The empty-reasoning skip is a `return` inside the method (it was a loop `continue` when inline).
- `commit_attempt_messages` is the single shared "build assistant message → (only if the attempt issued tool calls) execute tools + append the `ToolResult` user message + record the deferred-completion group" path, called by BOTH the mid-stream-retry branch and the terminal tool-call branch so the two cannot drift (the prior duplication was a real correctness hazard: a reconnected turn could build different history than a fresh one). Keep it **emission-free** — the retry path emits `AgentMessageCompleted` (partial text) + `StreamError` *around* the call; the terminal tool-call path emits nothing there. Reasoning is finalized at the call sites via `AttemptState::finalize_reasoning` (which consumes the delta buffer) *before* `commit_attempt_messages`, not inside it, so an attempt finalizes exactly once. Tool execution and the completion-group push stay gated on `saw_tool_call_this_attempt`; a no-tool retry must leave `stream_phase_completions` intact for the terminal no-tool branches.
- Subagent reclamation goes through `consume_child(agent_control, child) -> ThreadId` (releases in-memory resources via `release_consumed_child`, returns the id). It returns the id rather than taking `&mut consumed_children` so `drain_retained_subagent_completions` can call it inside each concurrent per-child future and still accumulate `consumed_children` sequentially after the `join_all` (a shared `&mut Vec` cannot cross those futures). Edge closing stays deferred to `close_consumed_child_edges` after `persist_turn_tail` succeeds — never on the cancel/early-return/persist-failure paths — so resume reaps a finished child whose result never became durable. See [[Rollout and Resume Persistence]].
- `AGENT_MAX_DEPTH` (= 8) lives in `agent::control` and is re-exported through `agent::mod`; `thread_manager::resume_child_thread` uses the constant instead of a hardcoded literal for both the depth check and the `max_depth` error field. `ContextManager` (`context_manager/history.rs`) is a thin `Vec<Message>` wrapper — the earlier `history_version` / `token_info` / `reference_context_item` scaffolding was unused and was removed; reintroduce it alongside real compaction/diffing rather than leaving dead fields.

## TUI File Change Display

- Successful `delete`, `edit`, and `write` tools can carry structured `FileChangeOutput` metadata through `ToolCallCompletedEvent.file_change`; core decodes this metadata before sending tool results back to the model so model-visible output remains the concise success message.
- Structured file-change output now also carries `fileChanges` for multi-file edit tools. Keep `fileChange` as a single-change compatibility alias for old persisted events and for single-change producers, but prefer the list when rendering or forwarding completed tool events.
- The TUI replaces a single completed file-mutating tool row with a patch transcript item and Codex-style diff summary; grouped file tool calls keep their group row and append the diff item so other entries are not lost.
- A successful `edit` completion may contain multiple file changes: the TUI replaces the tool row with the first patch item, appends subsequent patch items after it, and records every change in `recent_file_changes`.
- `FileChange::Update.move_path` is rendered as a move label (`old -> new`) in transcript diff summaries and recent-file inspector rows; rename-only edits may have an empty unified diff but should still be visible through that move label.
- TUI file-change diffs use `syntect` plus `two-face` grammars (`syntect-onig`, with syntect defaults disabled) for Codex-style syntax-colored content. Language detection is file-extension based and uses `move_path` for renamed updates; update hunks are highlighted as a whole hunk so multi-line parser state is preserved. Smooth keeps its existing summary/indent behavior and adds row-level add/delete background cues instead of carrying over Codex's full terminal-background/theme-picker palette stack.
- File-change transcript items render the full bounded diff/result in the transcript widget; the inspector should stay an overview surface (turn/mode/running tools/recent file changes) rather than the primary place to read edit/write/delete results.

## TUI Architecture

- `smooth-tui` state now flows through `UiModel::update(UiEvent) -> Vec<UiEffect>`; `App` is the async effect runner and renderer, and effect completions/failures re-enter the reducer by deterministic `EffectId`.
- TUI startup now opens a dashboard backed by `ThreadList`; `n` starts a new thread and `Enter` resumes the selected `ThreadListItem`. Resume hydration replays `ThreadResumeResponse.initial_messages` through the same protocol-event reducer path used for live events.
- Transcript storage is typed `TranscriptItem` data with stable item IDs and versions instead of dynamic `HistoryCell` trait objects. Cached row rendering is keyed by item ID, item version, width, and detail mode; resize eviction retains only current-width cache entries.
- Active assistant/reasoning streams remain separate mutable overlays and are committed into typed transcript items only when finalized, preserving the dedupe rules around deltas, completed events, late reasoning completions, and turn-completed fallbacks.
- `EventMsg::StreamError` is a transient reconnect notice: rollout intentionally does not persist it, agent status ignores it, and the TUI finalizes active assistant/reasoning overlays plus shows an info row without marking the turn stopped.
- Active streams are not in `render_cache` (they change every delta), so their wrapped output is memoized separately in `active_wrap`, keyed by `(width, active_version)`. Every write to the active lines goes through `set_active_*_lines`, which bumps `active_version`, so the memo can never go stale; this keeps the active block from being re-wrapped twice per frame (row count + visible) and on idle frames. The non-cached `append_active_lines` is now test-only.
- `TurnStart` effect context retains the submitted composer text so failed sends can restore the draft if the user has not typed a replacement while the effect was in flight.
- Transcript virtualization must count the same rows the renderer will draw: every transcript item type, including tool groups, goes through style-preserving wrapping before row counting, and active-stream separators are counted from total row state rather than currently visible rows.
- Transcript rows are pre-wrapped before virtualization; do not leave ratatui `Paragraph::wrap` enabled on the transcript pane, or any over-wide row that slips through (for example a diff gutter off-by-one) will be visually re-wrapped without being included in `max_scroll`.
- `AgentMessageCompleted` is the authoritative assistant transcript event; `AgentMessage` can follow with identical text from core/rollout replay and must be treated as a guarded fallback to avoid double-rendering live and resumed assistant messages.
- App-server subscriptions stay open for every started/resumed thread, so in-process `SessionEvent`s carry the subscription `ThreadId` and the TUI reducer ignores sourced protocol events that do not match `current_thread_id` before switching screens or mutating transcript state. Ask-user server requests are also scoped by their payload `thread_id`; inactive or invalid-thread requests are failed instead of opening overlays. Replay hydration still applies its `initial_messages` directly because the resume response is already thread-scoped.
- Keyboard scroll math should use the transcript pane's inner height, not a terminal-level approximation, so manual bottom scrolling matches what `render_transcript` actually draws.
- Row counting must also use the transcript pane's inner *width*, not `terminal_width`: in the split workspace the pane is only ~70% of the terminal, so wrapping (and thus row count and `max_scroll`) differs. `render_transcript` records the width it drew at (`transcript_inner_width`) and `max_scroll` reads it, so counting and drawing stay on the same width and the newest content stays reachable.
- The main TUI surfaces avoid full box borders so terminal selection stays copy-friendly. Prefer thin separators (`─`/`│`) between widgets, keep the transcript itself headerless/borderless, and make transcript row counting use the full rendered pane dimensions rather than subtracting border space.
- The workspace bottom stack places the input/composer above the status footer; command mode input also stays above the status separator so status remains the bottom-most line.
- Transcript wrapping is policy-split, not one-size: prose word-wraps (`wrap::wrap_line` breaks at spaces, hard-breaking only a word wider than the line, so words like "like" are not split), while code must stay column-faithful (`wrap::wrap_line_char`). Code blocks live inside assistant/reasoning Markdown, so `markdown_render` marks fenced code at the line level and `history_cell` routes those raw (pre-prefix) lines to char wrapping; inline code only colors spans, so even inline-code-only prose stays word-wrapped. Tool-group rows and diff body rows are char-wrapped too; diff body wrapping preserves syntax spans via the local styled-span wrapper in `diff_render`.
- Dashboard rendering never shows `QuestionPicker`; active ask-user requests switch to the workspace overlay, and hidden picker state must not intercept dashboard key routing. Inline file-change summaries wrap their header/path rows before truncation, keeping cached row counts aligned with rendered rows even for long paths.
- The TUI has dashboard/workspace screens plus Normal, Insert, Command, and Overlay modes. Ctrl-C remains an always-exit shortcut before mode or overlay dispatch; Visual mode and slash search remain intentionally unimplemented.
- The workspace inspector pane has session-local UI visibility (`UiModel::inspector_visible`, default true). Wide workspaces split Transcript/Inspector only when it is visible; hiding it gives Transcript the full body, removes Inspector from focus cycling, and `:focus inspector` intentionally re-shows it before focusing.
- The workspace composer is a cursor-aware `ComposerState` (extracted to `crates/tui/src/composer.rs`), not a raw `String`: Insert mode owns paste/editing keys, Normal mode keeps transcript scrolling, paste text is CRLF/CR-normalized into the composer, and composer cursor/render height use the same Unicode-width visual row calculation. The question picker's "Other" field reuses the same `ComposerState` (Left/Right/Home/End/Delete plus a reversed-cell cursor in `layout`); `handle_paste_event` routes paste to an open picker before the Insert-mode composer, flattening newlines/tabs to spaces because the field is single-line.
- Both `ServerRequest` arms (`AskUserQuestion` and `RequestPlanApproval`) reject a new request while any interactive overlay is pending, via `fail_server_request` (-32000, "another interactive request is already pending"). Without this, a second ask silently replaced the pending picker and orphaned its oneshot, hanging the first tool call until turn abort. Confirming the picker pushes a session-only `TranscriptItem::question_answers` summary row (`? question` / `→ answers`) before the `AnswerQuestion` effect; like `push_info` rows it is not replayed on resume.
- Insert-mode prompt submission is Ctrl+Enter on all platforms. Distinguishing Ctrl+Enter from a bare Enter (which inserts a newline) requires the kitty keyboard protocol, so TUI init pushes `KeyboardEnhancementFlags::DISAMBIGUATE_ESCAPE_CODES` and restoration pops it. Cmd/Super+Enter is intentionally *not* a submit chord: macOS terminals reserve Cmd for their own bindings (e.g. Ghostty maps `super+enter` to toggle-fullscreen), so it never reaches the app.

## TUI Structured Row Wrapping

- Structured rows (anything with a glyph, bullet, selection marker, or gutter) hang-indent their continuation rows under the row content. Assistant/reasoning prose intentionally keep flush-left (column-0) continuations via `wrap::wrap_line`/`wrap_line_char` — only the first row of a wrapped logical prose line carries the turn prefix, so do not "fix" them to hang-indent. **User messages are the exception**: `render_user_message` wraps the body to `width - 2` (reserving the 2-column gutter) *first*, then prefixes a blue `▌ ` gutter onto every wrapped row, so the gutter spans continuations (unlike the prefix-then-wrap prose path) and the user turn reads as a distinct block from the assistant's `•`.
- A user message is bracketed above and below by full-width muted (`DarkGray`) `─` rules (`user_separator`). The bottom rule is dropped when the message is the last thing rendered — the final transcript item with no active reasoning/assistant stream after it (`has_active_stream_lines`) — so the transcript never ends on a dangling rule, and the rule reappears as soon as a reply streams or commits. This is the one spot where rendered rows diverge from the cached `TranscriptItem::display_lines(width)`, so the drop is applied identically in `visible_transcript_lines`, `total_transcript_rows`, and the test-only `transcript_lines_uncached` (pop the trailing line, subtract one row) to keep row count and rendered rows aligned for scroll/virtualization.
- `wrap::wrap_line_char_hanging` (column-faithful) and `wrap::wrap_line_hanging` (word-aware) take a continuation indent measured in display columns: the first row uses the full width, continuation rows reserve `indent` columns and are prefixed with that many spaces. Callers compute the indent as the *display width* of the row's prefix (e.g. glyph(2) + tool name + space), never byte length — `wrap::display_width` is the shared helper. `wrap::truncate_display` cuts to a display-width budget and appends a single-column ellipsis.
- Tool args and diff summary/path rows use char-faithful hanging; info/error messages, the picker, the inspector, and dashboard errors use word-aware hanging. Diff *content* rows already preserved their gutter via `render_diff_line` (line number/sign re-emitted per wrapped chunk), so only the summary and path rows needed hanging indent; repeated `+`/`-` signs on wrapped diff lines are intentionally left as-is.
- `ToolCallGroupCell::display_lines(width)` wraps internally and returns final rows (rather than returning logical rows for a caller to wrap), so each row can carry its own computed hanging indent. Transcript scroll/virtualization stays consistent automatically because row count and rendered rows both come from the same `TranscriptItem::display_lines(width)` the render cache keys on (the sole exception is the last user message's dropped bottom rule, handled in lockstep across the three render paths as noted above). `info`/`error` are the only `Plain` transcript items; `Plain` carries a `hang_indent` (2) so the glyph-prefixed message keeps alignment when it wraps.
- The dashboard keeps a fixed two-rows-per-session invariant that its scroll math depends on (`dashboard_visible_item_count` = `(height - 4) / 2`). Long previews are display-width truncated with an ellipsis after the selection prefix + date + gap rather than wrapped, and the residual `Paragraph` `Wrap` was removed so an off-by-one truncation clips (contained to the row) instead of silently re-wrapping and breaking the count/footer.
- The ask-user picker's `render` and `desired_height(width)` share one `layout(width)` routine so their row counts cannot drift; `desired_height` returns the unclamped count and `render_workspace` caps the pane at 20 rows. `render` pins the header separator and footer/hint and windows only the option list (`window_start`) to keep the full active option block (cursor row + wrapped description + preview) visible; if the active block alone exceeds the window its top is pinned. The residual `Paragraph` `Wrap` was removed since every picker row is pre-wrapped. Inspector rows are likewise all pre-wrapped (including Turn/Mode/Thread) before its `Wrap` was removed, so long thread ids hang-indent instead of clipping.

## TUI File Change Safety

- Structured file-change tool output is a private transport and must only be decoded for successful final built-in `delete`/`edit`/`write` completions; arbitrary tool stdout and opaque stream outputs stay plain text to avoid spoofed diffs.
- File-change metadata is capped at 512 KiB; oversized diffs/new-file contents and unreadable/non-UTF8 existing files use `FileChange::Omitted` so the TUI can show counts/reason without carrying or rendering large/unavailable content.
- The TUI diff renderer caps rendered diff body lines at 1,000 and appends a truncation marker, protecting transcript recalculation and scroll performance.
- Omitted file changes must preserve the original operation (`add`, `delete`, or `update`) so large added files still render as Added and external JSON clients receive operation-aware metadata.
